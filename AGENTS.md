# Repository Guidelines

## 项目结构与内容组织
这是一个以 Markdown 为核心的个人知识库，不是传统代码项目。主要目录如下：

- `收集/`：原始输入，优先写入 `收集/Inbox.md`
- `日志/`：按天记录，文件名使用 `YYYY-MM-DD.md`
- `知识/`：长期知识笔记，目录结构遵循“`大类/主题/分组/文件`”
- `感悟/`、`总结/`、`展望/`：分别存放观点体会、复盘和计划
- `模板/`：统一模板来源
- `提示词/`：一次性整理或维护任务的提示词
- `索引/`：导航页，如 `索引/工作台.md`
- `.obsidian/`、`.vscode/`：本地工作区配置

图片等资源放在对应主题同级的 `资源/` 目录，不单独建立通用素材层。

## 构建、检查与常用命令
仓库没有构建流程，也没有自动化测试。日常维护以结构检查和内容校验为主：

- `rg --files`：快速查看当前仓库文件结构
- `sed -n '1,80p' 模板/知识笔记模板.md`：检查标准笔记格式
- `git diff --check`：检查多余空格和格式问题
- `git status --short`：确认只改了预期文件

修改链接、索引页或工作区配置后，建议在 Obsidian 中实际预览。

## 编写规范与命名约定
统一使用 Markdown，段落短、标题明确、列表直接。结构化笔记应保留 YAML frontmatter：

```yaml
---
title: 标题
type: knowledge
date: 2026-04-14
status: distilled
---
```

命名遵循现有约定：

- 日志：`YYYY-MM-DD.md`
- 总结或计划：`2026年总结.md`、`2026年计划.md`
- 感悟：简短主题名或日期型命名
- 知识笔记：直接使用主题名，避免“笔记”“总结”这类空词

## 校验要求
提交前至少做以下人工检查：

- frontmatter 字段是否完整且取值合理
- 新建或移动笔记后，内部链接是否仍然有效
- 新增知识笔记是否补上相关笔记或索引入口
- 不确定内容是否明确标记为 `待核实`

优先更新已有笔记，避免制造内容高度重复的新文件。

## 提交与合并请求规范
现有提交历史主要使用简短祈使句，例如 `Add ...`、`Update ...`、`Rename ...`。继续沿用这一风格，并保证一次提交只处理一类内容变更。

如果发起 Pull Request，建议包含：

- 本次改动涉及的目录范围，例如 `知识/操作系统` 或 `日志/2026`
- 如有目录调整或文件迁移，说明原因
- 仅在 `.obsidian/` 视图或工作区设置变化时附截图

除非明确要求，不要删除 `收集/` 和 `日志/` 中的原始记录。
