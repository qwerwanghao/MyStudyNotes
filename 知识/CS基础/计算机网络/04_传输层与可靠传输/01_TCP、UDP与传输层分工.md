# TCP、UDP与传输层分工

> 这篇笔记要回答：**为什么网络不只提供一种“发送数据”方式；TCP 和 UDP 到底各自在解决什么问题，为什么不是简单的“一个高级一个低级”。**

## 为什么传输层要单独存在

光有 IP 只能保证：
- 包朝着目标主机去

但还没解决：
- 应该交给目标主机上的哪个进程
- 数据乱序、丢失了怎么办
- 发送方和接收方节奏不一致怎么办

所以传输层的核心是：
> **把“主机到主机”推进成“进程到进程”的通信能力。**

## 先建立一个核心心智模型

### TCP 更偏“可靠有序的字节流服务”，UDP 更偏“轻量的数据报投递”
这两者不是谁替代谁，
而是服务模型不同：
- TCP 更强调可靠、有序、连接语义
- UDP 更强调简单、低额外约束、应用自己决定控制策略

## 端口为什么重要

端口的意义不是数字本身，
而是：
- 让一台主机上的不同应用 / 进程能被区分

所以 IP 解决“到哪台机器”，
端口解决“到机器里的哪个服务”。

## 一个最小过程案例：为什么视频通话和网页请求未必偏好同一种传输方式

网页请求常常更看重：
- 完整性
- 顺序性
- 可靠交付

实时音视频则可能更看重：
- 低延迟
- 丢一点也能继续播放
- 不希望旧包迟到反而拖累当前体验

这就是为什么它们可能偏好不同传输模型。

## 常见误区

### 误区 1：UDP 就是不可靠，所以没用
错误。UDP 只是把更多控制权留给应用，不等于不能构建高价值系统。

### 误区 2：TCP 一定更快，因为功能更全
不对。功能更全往往伴随更高约束与额外开销。

### 误区 3：端口只是服务编号，和通信模型关系不大
不完整。端口是进程级复用与分发的关键入口。

## 最小例题

### 例题 1：为什么只有 IP 还不足以支撑应用通信？
答案要点：
- 因为还要区分目标主机上的具体进程
- 还要解决可靠性和传输语义问题

### 例题 2：为什么“低延迟”场景不一定偏爱 TCP？
答案要点：
- 因为严格可靠和有序有时会带来额外等待
- 某些实时场景更愿意接受部分丢失而不要额外拖延

## 对比题

### TCP vs UDP，最本质的差异是“功能多少”，还是“服务模型不同”？为什么？

## 参考答案

### 例题 1 参考答案
只有 IP 不够，因为它只解决主机级寻址，不解决进程复用、可靠顺序和传输控制问题。传输层正是为这些问题存在。

### 例题 2 参考答案
低延迟场景不一定偏爱 TCP，因为 TCP 的可靠有序语义可能引入等待与重传成本，而某些实时业务更重视当前时刻的时效性。

### 对比题参考讲法
TCP 和 UDP 的关键区别不是“一个功能全、一个功能少”，而是它们提供的服务模型不同，适用业务目标也不同。

## 常见误答

- “UDP 就是过时协议。”
- “TCP 一定比 UDP 更先进所以应优先使用。”
- “有了 IP 地址就不需要端口概念。”
