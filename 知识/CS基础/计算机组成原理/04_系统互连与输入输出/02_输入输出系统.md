# 输入输出系统

> 这篇笔记要解决的问题是：**CPU 和内存并不能直接替代设备工作，那么机器到底怎样和键盘、磁盘、网卡、显示器等外部世界连接起来？**

## 为什么 I/O 是单独的一整块系统

如果只有 CPU 和内存，机器顶多只能在内部算。

真正让计算机变成“可用系统”的，是它能够：
- 接收输入
- 输出结果
- 和外界设备交换数据

但这里马上会出现一个大矛盾：
- CPU 很快
- 设备常常慢很多，行为也不统一

所以 I/O 系统的核心任务，就是协调：
- 速度差异
- 工作方式差异
- 数据格式差异
- 控制方式差异

## 先建立一个核心心智模型

### I/O 系统本质上是“主机系统”和“外设世界”的翻译与协调层
你可以先把 I/O 看成三个角色：

1. **设备本体**
   - 真正和外界交互的物理对象
2. **接口 / 控制器**
   - 负责把设备能力转换成系统能管理的形式
3. **CPU / 内存 / 总线**
   - 负责主机侧的控制与数据协同

所以 I/O 并不是“设备自己干活”，而是整个系统一起完成一次对外协作。

## 为什么设备必须分类理解

设备差异很大，主要体现在：
- 速度
- 数据粒度
- 是否连续流式工作
- 是否支持随机访问

### 字符设备
更像逐个字符或小单位流式传输。

### 块设备
更像按块批量传输，常见于存储类设备。

### 网络类设备
更像按包或帧进行通信。

理解分类的目的不是背名词，而是知道：
- 设备类型不同，系统管理方式也会不同。

## 接口 / 控制器为什么不可缺

CPU 不能直接裸连所有设备细节。

所以接口或控制器要承担：
- 数据缓冲
- 时序协调
- 控制信号转换
- 状态报告
- 必要时触发中断

这层非常关键，因为：
> 它让 CPU 不必理解每个设备的底层物理细节，而能通过统一方式去管理 I/O。

## 为什么 I/O 方式会有层次差别

这部分是全篇最关键的理解点之一。

## 1. 程序查询方式（轮询）
CPU 反复问设备：
- 你好了吗？
- 现在能传了吗？

优点：
- 简单直接

问题：
- CPU 会浪费大量时间等待

这适合：
- 很简单的场景
- 很低速或控制逻辑容易实现的设备

### 一个轮询直觉例子
假设程序想从键盘读一个字符：
- CPU 不停检查设备状态位
- 如果键盘还没准备好，就继续等
- 一旦准备好了，再把字符读走

这说明轮询的问题不在“做不到”，而在：
- **CPU 被迫把很多时间花在等待上。**

## 2. 中断方式
设备准备好后，不再等 CPU 一直查询，而是主动通知 CPU。

优点：
- CPU 不必一直空等
- 系统响应更灵活

问题：
- 中断太频繁时，CPU 仍可能被打断得很碎

这部分会和 [中断、异常与上下文切换](../03_存储、指令与CPU/07_中断、异常与上下文切换.md) 强相关。

### 一个中断直觉例子
假设网卡收到一个数据包：
- 它不需要 CPU 每毫秒都来问“包到了没”
- 它可以等包真到了，再发中断通知 CPU

这样 CPU 就能：
- 平时干别的事
- 有事件时再回来处理

## 3. DMA 方式
当数据量更大时，如果还让 CPU 一点点搬运，会太浪费。

于是 CPU 只负责：
- 配置传输参数
- 启动 DMA
- 等待完成通知

真正搬数据的事交给 DMA 控制器。

这就是为什么 DMA 特别适合高速设备和大批量传输。

这部分我已经单独拆成 [DMA、设备控制器与I-O协同](03_DMA、设备控制器与I-O协同.md)。

### 一个 DMA 直觉例子
假设磁盘要把一大块数据读到内存里：
- 如果由 CPU 一字一字搬，会非常浪费
- 更好的做法是 CPU 先下命令
- 由 DMA 控制器负责真正搬运
- 结束后再通知 CPU

这个例子最能说明：
- **CPU 应该负责控制，不该负责当搬运工。**

## 为什么 I/O 不只是“设备章节”

因为 I/O 会同时影响：
- CPU 是否被频繁打断
- 总线是否被占用
- 内存是否被频繁访问
- 缓冲区是否足够
- 数据吞吐是否稳定

所以 I/O 不是外围小知识，而是整机性能和系统设计的重要组成部分。

## 常见误区

### 误区 1：I/O 慢只是因为设备硬件慢
不完全是。很多瓶颈来自：
- 协调方式
- 中断频率
- 总线争用
- 缓冲策略

### 误区 2：轮询一定很差
也不绝对。在很简单、很低速、很确定的场景下，轮询可能足够直接。

### 误区 3：DMA 只是“更快的 I/O”
不止如此。它本质上改变了“谁负责搬数据”的分工。

## 读完这篇后，你应该能回答的问题

- 为什么机器必须有单独的 I/O 系统，而不能只靠 CPU 和内存？
- 轮询、中断、DMA 三种方式本质上差在哪？
- 为什么 I/O 会影响 CPU、总线和内存三边的协同？
- 为什么“设备准备好了”这件事，不能总靠 CPU 自己一直盯着？

## 程序员最该带走的几个直觉

### 1. CPU 不该一直当搬运工
轮询和 CPU 亲自搬数据通常效率不高。

### 2. 设备和主机之间永远有“速度差”和“语义差”
I/O 系统的存在就是为了协调这种差异。

### 3. 中断适合事件通知，DMA适合大批量传输

### 4. I/O 性能问题经常不是“设备慢”这么简单，而是整条协作链出了问题

## 本章小结

- I/O 系统的核心任务，是协调主机系统和外部设备之间的差异
- 设备、接口 / 控制器、主机系统共同构成 I/O 协作链
- 轮询、中断、DMA 分别代表不同层次的 I/O 管理方式
- I/O 不是边缘知识，它会直接影响整机性能、响应和系统设计

## 最小例题

### 例题 1：键盘输入更适合哪种方式，为什么？
更适合：
- 轮询或中断中的较轻量方式（具体实现看系统）

关键原因：
- 数据量小
- 事件型触发更常见
- 不值得让 CPU 为极小数据做复杂大批量搬运设计

### 例题 2：磁盘搬大块数据为什么更适合 DMA？
因为：
- 数据量大
- 如果由 CPU 一点点搬，开销太大
- DMA 能把“控制”和“搬运”分工开

## 对比题

### 轮询 vs 中断 vs DMA，最本质的三问是什么？
请分别回答：
- 谁在等？
- 谁在搬？
- 谁在通知？

## 参考答案

### 例题 1 参考答案
键盘输入通常更适合轻量事件式处理：
- 数据量小
- 不是持续大批量流
- 重点在“有事件时响应”，不是持续高速搬运
因此轮询或中断更自然，DMA 往往没有必要。

### 例题 2 参考答案
磁盘搬大块数据更适合 DMA，因为：
- 数据量大
- CPU 不应该为大批量传输亲自做逐字搬运
- DMA 让 CPU 负责发命令，控制器负责真正搬运

### 对比题参考讲法
- **轮询**：CPU 自己一直盯着设备状态，所以 CPU 在等、CPU 也要参与搬。
- **中断**：设备准备好后通知 CPU，CPU 不必一直等，但处理时仍需 CPU 介入。
- **DMA**：CPU 先配置，真正搬运主要由 DMA 控制器负责，完成后再通知 CPU。

## 常见误答

- 误答 1：把 DMA 简化成“更快的中断”。
- 误答 2：把轮询说成“完全没用”，忽略它在简单设备和简单场景下仍然直接有效。
- 误答 3：把 I/O 性能问题只归咎于设备慢，而忽略协调链路。

## 能力判定

- **入门**：能区分轮询、中断、DMA 三种 I/O 方式
- **理解**：能说清三者在“谁在等 / 谁在搬 / 谁在通知”上的差异
- **熟练**：能根据设备场景判断哪种方式更自然
- **迁移**：能把 I/O 问题和 CPU、总线、内存协同联系起来

- 说清为什么 CPU 不该一直当搬运工
