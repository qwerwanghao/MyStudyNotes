# Cache、局部性与存储层次

> 这篇笔记专门回答一个程序员最常见但最容易模糊的问题：**为什么 CPU 已经这么快了，程序还是会被“访存”拖慢？**

- 后续延伸：读完后接 [虚拟存储、地址转换与页表](05_虚拟存储、地址转换与页表.md) 与 [多核、缓存一致性与存储模型](../05_性能扩展与并行处理/02_多核、缓存一致性与存储模型.md)

## 为什么这个专题必须独立出来

如果只知道“有 Cache”但不知道：
- 什么是局部性
- 什么是 cache line
- 什么是冲突失效
- 写策略为什么影响性能

那你在看：
- 数组为什么比链表更 cache-friendly
- 为什么循环交换顺序会快很多
- 为什么 false sharing 会拖慢多线程

这些问题时，就只能记结论，没法真的理解。

## 先建立一个核心心智模型

### 存储层次不是“容量表”，而是“延迟交易结构”
从寄存器、L1/L2/L3 Cache、主存，到外存，背后是一个基本矛盾：

- **越快的存储，越贵、越小**
- **越大的存储，越慢、越便宜**

所以现代计算机并不是直接让 CPU 面向“大内存”工作，而是让数据分层靠近处理器。

### Cache 能成立，靠的是局部性

#### 时间局部性
刚访问过的数据，短时间内还可能再访问。

典型例子：
- 循环里的计数变量
- 热点函数里的局部变量
- 频繁读取的配置项

#### 空间局部性
访问了某个地址后，附近地址也很可能很快被访问。

典型例子：
- 顺序遍历数组
- 扫描字符串
- 线性读结构体数组

> 你可以把 Cache 理解成：**不是在缓存“某个变量”，而是在赌“你接下来还会用到这片附近的数据”。**

## Cache 的基本组成

### 1. Cache line（缓存行）
Cache 与主存交换数据时，不是一个字节一个字节搬，而是一整块搬。

这整块就叫 cache line。

理解重点：
- 局部性的“空间单位”就是 cache line
- 一次 miss 不只是取一个值，而是把附近一段也一起带进来

### 2. Tag / Index / Offset
CPU 访问某个内存地址时，通常要把地址拆成：
- **offset**：块内偏移，表示数据在 cache line 里的位置
- **index**：定位到哪一组 / 哪一行
- **tag**：判断这一行里是不是我要的数据

> 这套拆分就是“地址如何映射到 Cache”的核心。

## 三种常见映射方式

### 直接映射
- 每个主存块只能去唯一一个位置
- 实现简单，硬件成本低
- 但容易发生冲突失效

### 全相联映射
- 主存块可以放到任意位置
- 冲突少，命中率更高
- 但查找成本高，硬件复杂

### 组相联映射
- 折中方案，也是最常见的工程选择
- 先定位到某一组，再在组内查找

## Cache miss 的三种来源

### 1. 强制失效（compulsory miss）
第一次访问这块数据，之前不可能命中。

### 2. 容量失效（capacity miss）
工作集太大，Cache 装不下。

### 3. 冲突失效（conflict miss）
明明总容量可能够，但若干地址映射到同一组，互相挤掉了。

> 这三类失效，对应三类优化思路：
> - 首次访问无法避免，只能做预取或批量化
> - 容量问题要缩小工作集
> - 冲突问题要调整数据布局和访问模式

## 写策略为什么也很关键

### 写直达（write-through）
- 每次写 Cache，同时写主存
- 一致性更直观
- 但写流量大

### 写回（write-back）
- 先只写 Cache
- 被替换时再回写主存
- 性能更好，但控制更复杂

### 写分配 / 非写分配
当写 miss 发生时：
- **write-allocate**：先把块拉进 Cache 再写
- **no-write-allocate**：直接写下层，不进 Cache

这会影响：
- 写密集工作负载的性能
- 多核下一致性开销

## 你真正该带走的程序员视角

### 1. 顺序访问通常比跳跃访问更友好
因为顺序访问更能利用空间局部性。

### 2. 数组通常比链表更 cache-friendly
因为数组连续，链表节点可能分散在内存各处。

### 3. 热数据应该尽量小而集中
让工作集尽量留在更近的 Cache 层。

### 4. “算法复杂度一样”不代表跑得一样快
同样是 O(n)，不同访问模式的实际耗时可能差很多。

## 读这一篇时最值得不断追问的问题

- 这个程序访问的是“一个值”，还是“一片连续区域”？
- 访问模式是顺序的、局部复用的，还是离散跳跃的？
- 当前瓶颈是算得慢，还是取数据慢？
- 如果多核同时访问同一块数据，会不会带来一致性开销？

## 最小例题

### 例题 1：为什么顺序遍历数组通常比随机跳跃访问更容易跑得快？
答案要点：
- 顺序遍历更能利用空间局部性
- 一次 cache line 取回后，附近元素更可能马上被用到
- 随机跳跃更容易浪费缓存行并增加 miss

### 例题 2：为什么两个逻辑复杂度都为 O(n) 的程序，实际性能可能差很多？
答案要点：
- 算法复杂度没有描述访存模式
- 一个可能顺序扫数组，一个可能频繁追链表
- 后者更容易被 Cache miss 拖慢

## 对比题

### Cache vs 主存 vs TLB，最应该先区分什么？
请先回答：
- 它们分别在缓存什么？
- 解决的是数据访问慢、地址翻译慢，还是容量问题？

## 参考答案

### 例题 1 参考答案
顺序遍历数组更快，核心原因是：
- 数组元素连续
- Cache 一次按 line 取回一整块数据
- 当前元素被访问后，后续相邻元素大概率已经在缓存里
这正是空间局部性的直接体现。

### 例题 2 参考答案
两个 O(n) 程序也能差很多，因为：
- O(n) 只说明增长阶，不说明访问模式
- 连续访问更容易命中 Cache
- 离散访问会让 CPU 频繁等待数据
所以“算得一样多”不代表“取数据一样顺”。

### 对比题参考讲法
- Cache：缓存热点数据块，主要解决 CPU 等数据太久
- 主存：主要工作区，容量大但更慢
- TLB：缓存地址翻译结果，不是缓存普通程序数据

## 常见误答

- “Cache 就是更小更快的内存，理解到这就够了。”
- “只要算法复杂度一样，访问模式对性能影响不大。”
- “TLB 和 Cache 只是叫法不同，本质完全一样。”
