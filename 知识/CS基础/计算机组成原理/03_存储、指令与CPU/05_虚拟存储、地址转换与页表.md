# 虚拟存储、地址转换与页表

> 这篇笔记的目标是把一句非常重要的话讲透：**程序看到的内存，并不是机器真正“裸露出来”的物理内存。**

## 为什么这个主题值得独立成篇

很多人知道：
- 有虚拟地址
- 有物理地址
- 有页表
- 有 TLB

但不知道它们为什么非要一起出现。

真正需要理解的是：
1. 为什么要做地址转换
2. 地址转换为什么不能太慢
3. 为什么页表和 TLB 是一整套方案
4. 为什么“缺页”既是硬件现象，也是系统设计问题

## 它在计组路线里的位置

- 前置：读过 [存储系统](01_存储系统.md) 会更容易
- 推荐联动：和 [Cache、局部性与存储层次](04_Cache、局部性与存储层次.md) 一起看
- 后续会连接到操作系统主题，但在计组里先把硬件视角打通

## 先看最核心的问题：为什么不能直接用物理地址

如果程序直接操作物理地址，会带来几个大问题：

### 1. 地址空间不独立
不同程序会互相踩内存。

### 2. 程序装载不灵活
同一程序如果必须放在固定物理位置，装载和重定位都会很麻烦。

### 3. 安全与保护困难
系统很难限制用户程序“不准访问哪里”。

### 4. 内存利用率低
程序可能只活跃使用一部分数据，但又必须整体驻留。

所以，程序看到的是**虚拟地址空间**，硬件和系统再把它翻译成物理地址。

## 一个程序访问内存时，实际发生了什么

从程序视角看，只是访问某个地址。

从机器视角看，背后通常经历：
1. CPU 发出虚拟地址
2. 先查 TLB，看是否已有最近的地址转换缓存
3. 如果 TLB 命中，直接得到物理页框号
4. 如果 TLB 未命中，则查页表
5. 若页表项有效，再得到物理地址
6. 若页不在内存中，则触发缺页异常，由系统介入

> 所以“访存”不只是取数据，有时先得完成一轮地址翻译。

## 一个最小过程案例：访问某个虚拟地址时发生了什么

假设程序现在要访问虚拟地址 `0x12345678`。

从抽象上看，可以把过程想成：

1. 先把地址拆成：
   - 虚拟页号
   - 页内偏移
2. CPU 先拿虚拟页号去查 TLB
3. 如果 TLB 命中，就能迅速得到物理页框号
4. 再把“物理页框号 + 页内偏移”拼成物理地址
5. 最后再去访问真正的数据

如果 TLB 没命中：
- 就得去查页表
- 查到之后还可能更新 TLB

如果页表还显示“这页不在内存”：
- 就会触发缺页

这个例子最重要的价值是告诉你：
> 一次“普通访存”背后，可能先经历一整套地址翻译流程。

## 页、页框、页表：三个容易混但必须分清的概念

### 页（page）
虚拟地址空间被切成一页一页。

### 页框（frame）
物理内存被切成同样大小的一块一块。

### 页表（page table）
记录“虚拟页号 -> 物理页框号”的映射关系。

重点不是定义，而是这个思想：
- 程序看到的是逻辑上的连续空间
- 物理内存里可以是离散分布的
- 页表负责把“连续的感觉”和“离散的现实”接起来

## TLB 为什么这么重要

如果每次访存都要先去主存里查页表，再去主存里取真正数据，那代价会非常高。

所以处理器会用 **TLB（Translation Lookaside Buffer）** 来缓存最近用过的页表项。

这和 Cache 的思想很像：
- Cache 缓存的是“数据块”
- TLB 缓存的是“地址翻译结果”

> 也就是说：一条内存访问可能同时受 **Cache 命中率** 和 **TLB 命中率** 影响。

## 缺页到底是什么

缺页不是“程序崩了”，而是：
- CPU 发现页表显示这页当前不在主存
- 触发异常
- 操作系统介入，把页从磁盘调入内存
- 更新页表
- 再恢复执行

这说明虚拟存储不只是“地址转换”，更是一种：
- 用磁盘扩展内存可见空间
- 让热数据留在主存
- 让冷数据暂时待在外存

的整体机制。

## 一个缺页的现实直觉例子

假设你打开一个很大的程序：
- 并不是所有代码和数据都会一瞬间全被放进最快层里
- 某些部分可能在真正访问时才被调入

如果某次访问碰到了尚未驻留的页，
系统就得：
- 找到对应数据
- 从外存调进内存
- 更新页表
- 恢复执行

所以缺页真正贵的地方在于：
- **你本来只是想读一个数据，却被迫触发了一次跨层搬运。**

## 常见误区

### 误区 1：虚拟内存就是“内存变大了”
不够准确。它更核心的意义是：
- 地址空间抽象
- 保护隔离
- 延迟调页
- 灵活装载

### 误区 2：TLB 是某种特殊内存
更准确地说，它是地址翻译结果的高速缓存。

### 误区 3：缺页就是程序错误
不一定。缺页既可能是正常的按需调页过程，也可能是非法访问导致的异常。

## 程序员最该理解的几个关键后果

### 1. 连续访问通常更友好
因为页、cache line、TLB 都偏爱局部性强的访问模式。

### 2. 大工作集会同时冲击 Cache 和 TLB
数据一大，不只是缓存压力大，地址转换压力也会上升。

### 3. 随机访存常常很贵
因为它既可能导致 cache miss，也可能导致 TLB miss，甚至触发缺页。

### 4. 虚拟内存不是“白送的大内存”
它换来的是灵活性和保护，但真正缺页时成本很高。

## 最小例题

### 例题 1：一次虚拟地址访问最少会拆成哪两部分？
以虚拟地址访问为例，最基本要拆成：
- 虚拟页号
- 页内偏移

然后：
- 先查 TLB / 页表确定页框
- 再和页内偏移拼回物理地址

### 例题 2：为什么一次访存可能突然变得很慢？
可能路径是：
- TLB miss
- 页表查找
- 甚至页不在内存里，触发缺页
- 系统不得不从外存调页

## 对比题

### Cache vs TLB，最本质的区别是什么？
请不要只说“一个缓存数据，一个缓存地址”。
更好的答案应该说明：
- 它们分别服务哪一层问题
- 它们都为什么和局部性有关
- 为什么一次访存会同时受它们影响

## 参考答案

### 例题 1 参考答案
一次虚拟地址访问至少要区分：
- 虚拟页号：用来查映射
- 页内偏移：用来定位页内具体字节

查到物理页框后，再和页内偏移组合成物理地址。

### 例题 2 参考答案
访存变慢的常见原因不是“CPU 变笨了”，而是访存链条拉长了：
- 先可能 TLB miss
- 再可能页表查找
- 最坏还可能缺页，从外存调页
所以一次普通读操作可能被放大成跨层访问。

### 对比题参考讲法
- **Cache**：缓存的是数据内容，目标是让数据更快被 CPU 拿到。
- **TLB**：缓存的是地址翻译结果，目标是让“虚拟地址 → 物理地址”的转换更快。
- 两者都利用局部性，但服务的层不同：
  - Cache 服务“数据访问层”
  - TLB 服务“地址转换层”
- 一次访存可能先受 TLB 命中率影响，再受 Cache 命中率影响。

## 常见误答

- 误答 1：把 TLB 说成“特殊内存”，却说不清它缓存的是翻译结果。
- 误答 2：把虚拟内存简单理解成“内存变大了”。
- 误答 3：把缺页一律理解成“程序出错”。
