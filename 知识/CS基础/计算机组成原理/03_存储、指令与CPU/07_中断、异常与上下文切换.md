# 中断、异常与上下文切换

> 这篇笔记要打通一个非常关键的链条：**程序为什么不是“一路顺着往下跑”，而是会被设备、错误、系统调用和调度器打断。**

## 为什么这个主题一定要单独讲

在 CPU 和 I/O 笔记里，中断往往只是一个章节，但真正要形成完整系统观，需要单独区分：
- 中断（interrupt）
- 异常（exception）
- 陷入 / trap / 系统调用
- 上下文切换（context switch）

如果这几件事混在一起，后面学操作系统会非常乱。

## 一句话先分清四个概念

### 中断
通常来自 CPU 外部，例如：
- 定时器
- 键盘
- 网卡
- 磁盘

特点：
- 往往是异步发生
- 和当前指令流不是同一个来源

### 异常
通常由当前执行指令触发，例如：
- 除零
- 非法指令
- 页故障
- 访问越权

特点：
- 与当前指令直接相关
- 通常是同步发生

### 陷入 / trap / 系统调用
本质上是“程序主动请求系统介入”的一种受控转移。

比如：
- 请求打印
- 请求分配内存
- 请求文件读写

### 上下文切换
不是“发生了某种事件”的名字，而是：
- 为了切换执行主体
- 保存当前状态
- 恢复另一个执行状态

的过程。

## 为什么不能总靠轮询

如果 CPU 一直靠轮询问设备：
- 你准备好了吗？
- 你准备好了吗？
- 你准备好了吗？

那大量时间都会浪费在无意义检查上。

所以现代系统更常用：
- 设备准备好了再发中断
- CPU 先去干别的
- 需要时再响应

这也是中断机制的现实价值：
- **让 CPU 不必被慢设备拖着空转。**

## 一次中断 / 异常发生时，系统通常做什么

1. 识别事件来源
2. 保存当前执行现场
3. 切换到特权态或中断处理上下文
4. 根据中断向量 / 异常号找到处理入口
5. 执行服务程序
6. 恢复现场，返回原程序或切换到新任务

## 什么叫“保存现场”

最核心的是保存当前执行所必需的信息，例如：
- 程序计数器 PC
- 通用寄存器
- 状态寄存器 / 标志位
- 某些体系结构下的控制寄存器信息

如果不保存这些内容，程序被打断后就回不到原来位置。

## 上下文切换为什么贵

很多人以为“切换一下任务”只是换个指针，其实代价往往很高，因为它可能牵涉：
- 保存 / 恢复寄存器
- 改变地址空间
- 刷新或污染 Cache / TLB
- 破坏流水线与预测状态

> 所以并不是“线程越多越好”，切换本身就是成本。

## 这一篇和操作系统的边界

这篇先从**硬件与执行流控制**的角度理解：
- 事件怎样打断当前执行
- 处理器怎样转入处理程序
- 为什么需要保存上下文

以后到了操作系统主题，会再从：
- 调度
- 内核态 / 用户态
- 系统调用路径
- 缺页异常处理

这些更系统的软件视角再看一次。

## 最容易混淆的几个点

### 中断 ≠ 异常
- 中断多来自外部设备或时钟
- 异常多由当前指令触发

### 系统调用 ≠ 普通函数调用
普通函数调用仍在同一权限层里跳转；系统调用意味着进入受保护的系统服务边界。

### 上下文切换 ≠ 中断本身
中断可能导致上下文切换，但两者不是同义词。

## 适合带着哪些问题来读

- CPU 为什么要允许“当前执行流被插队”
- 页故障为什么属于异常，而不是普通错误提示
- 系统调用为什么本质上是一种受控陷入
- 为什么线程切换会伤害缓存和执行连续性

## 最小例题

### 例题 1：键盘按下一个键后触发 CPU 处理，这更像中断、异常，还是上下文切换？
答案要点：
- 更像外部设备引发的中断
- 是否发生上下文切换要看系统后续是否决定换任务
- 它本身不等于异常

### 例题 2：程序执行除零指令时发生的事件，为什么更接近异常而不是中断？
答案要点：
- 原因来自当前正在执行的指令
- 与当前执行流同步相关
- 不是外部设备异步打断

## 对比题

### 中断 vs 异常 vs 系统调用 vs 上下文切换，最本质的差别是什么？
请回答：
- 事件来源是谁？
- 是被动打断还是主动请求？
- 有没有一定导致执行主体切换？

## 参考答案

### 例题 1 参考答案
键盘事件更像中断，因为：
- 来源是外部设备
- 它通常异步到来
- CPU 需要响应事件
但它不必然意味着上下文切换；系统也可能处理完后继续回到原来的程序。

### 例题 2 参考答案
除零更像异常，因为：
- 它由当前那条指令直接触发
- 与当前执行流同步相关
- 不是设备或时钟从外部插进来的事件

### 对比题参考讲法
- 中断：多来自外部设备 / 时钟，偏异步
- 异常：多由当前指令触发，偏同步
- 系统调用：程序主动请求进入系统服务边界
- 上下文切换：保存并恢复不同执行主体状态的过程，不是某一类事件本身

## 常见误答

- “只要程序被打断，就都叫中断。”
- “系统调用就是普通函数调用，只是函数更特殊。”
- “一发生中断，就一定立刻切到另一个线程。”

## 能力判定

### 入门
- 能区分事件来源和同步 / 异步差异

### 理解
- 能说明为什么页故障是异常、设备就绪是中断

### 熟练
- 能把中断、异常、系统调用和调度切换放到同一条执行流图里解释清楚

- 你能解释为什么上下文切换不是“事件名字”，而是一个过程吗？
- 你能说明为什么线程切换会伤害 Cache 和执行连续性吗？
