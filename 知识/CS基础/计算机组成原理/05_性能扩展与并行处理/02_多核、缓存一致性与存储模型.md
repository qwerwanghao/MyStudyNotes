# 多核、缓存一致性与存储模型

> 单核时代最核心的问题是“怎样让一条执行链更快”，多核时代最核心的问题则变成：**怎样让多个核心在看到“同一份内存”时，不至于互相打架。**

## 为什么这个专题必须补上

当前计组路径里已经有 [并行处理系统](01_并行处理系统.md)，但如果没有这一篇，通常会缺少课程级理解中的关键一层：
- 多核共享内存为什么难
- 缓存一致性和内存一致性不是一回事
- false sharing 为什么会让代码“逻辑正确但跑得很慢”
- 为什么并发程序的性能和正确性都受底层内存模型影响

## 先分清两个极易混淆的概念

### 缓存一致性（cache coherence）
关心的是：
- 如果多个核心缓存了同一内存位置
- 其中一个核心修改了值
- 其他核心什么时候看到更新

它回答的是：**同一个地址的多个副本是否一致。**

### 存储模型 / 内存一致性（memory consistency model）
关心的是：
- 多个内存操作在不同核心看来，顺序是否一致
- 程序员能对“写入先后”做什么假设

它回答的是：**多个读写动作的可观察顺序是什么。**

> coherence 解决“同一位置多个副本”的问题；consistency 解决“多个操作顺序”的问题。

## 为什么多核一定会遇到这个问题

单核里，Cache 更多是性能优化。

多核里，每个核心都可能有自己的私有 Cache，于是会出现：
- 核心 A 改了变量 x
- 核心 B 还拿着旧的 cache line
- 两边都觉得自己看到的是“正确值”

如果没有一致性协议，共享内存就会失控。

## 一致性协议最核心的直觉

虽然具体协议很多，但直觉上都在做几件事：
- 标记某个 cache line 当前是什么状态
- 某核心写入前，先让其他核心的旧副本失效或更新
- 保证最终不会长期存在多个互相矛盾的版本

常见入门关键词：
- Modified
- Exclusive
- Shared
- Invalid

理解这些状态，不是为了死记协议图，而是为了看懂：
- 为什么一次写操作会牵动别的核心
- 为什么共享写很贵

## false sharing 为什么这么坑

两个线程明明改的是不同变量，却仍然互相拖慢。

原因往往不是变量逻辑冲突，而是：
- 它们碰巧落在同一个 cache line 里
- 一个核心改这行时，另一个核心的副本就要失效
- 于是来回“打架”

> 所以多线程优化时，真正的竞争对象有时不是变量，而是 **cache line**。

## 存储模型为什么会影响程序正确性

程序员直觉里常常默认：
- 先写 A
- 再写 B
- 别人就应该先看到 A，再看到 B

但现实里：
- 编译器可能重排
- CPU 可能乱序执行
- 写入传播到其他核心也可能不是同步瞬时完成

于是并发代码如果没有同步原语，就可能出现“逻辑上以为不可能，实际上会发生”的结果。

## 单核优化和多核优化的一个关键区别

### 单核优化
更多在想：
- 流水线能不能跑满
- Cache 命中率高不高
- 分支预测稳不稳

### 多核优化
除了上面这些，还必须想：
- 共享数据会不会导致一致性流量
- 锁竞争会不会太重
- 数据布局会不会导致 false sharing
- 同步点是不是太频繁

## 程序员最应该带走的结论

### 1. 多核不是“单核 x N”这么简单
共享内存和一致性使问题复杂得多。

### 2. 共享写通常最贵
读共享常常还能接受，写共享会触发更多协调开销。

### 3. 数据布局会影响并发性能
即使算法没错，cache line 布局也可能让程序变慢。

### 4. 不理解存储模型，就很难真正写对并发代码
高级语言帮你抽象了一部分，但底层约束并没有消失。

## 最适合带着哪些问题来读

- 为什么多核会让共享内存问题变难
- coherence 和 consistency 到底差在哪
- false sharing 为什么会让不同变量彼此干扰
- 为什么“没有加锁但测起来没问题”并不能说明代码正确

## 最小例题

### 例题 1：为什么两个线程修改的是不同变量，也可能彼此拖慢？
答案要点：
- 不同变量可能落在同一个 cache line
- 一个核心写入会让另一个核心的副本失效
- 这就是 false sharing

### 例题 2：为什么“我明明先写 A 再写 B”不一定意味着另一个核心也一定先看到 A 再看到 B？
答案要点：
- 编译器和 CPU 可能重排
- 写入传播到其他核心也可能有延迟
- 这属于存储模型要约束的问题

## 对比题

### cache coherence vs memory consistency，最本质分别回答什么问题？
请回答：
- 它们关心的是“同一地址的多个副本”，还是“多个操作的可观察顺序”？
- 它们分别更偏性能协同还是程序语义约束？

## 参考答案

### 例题 1 参考答案
不同变量也会彼此拖慢，因为：
- 多线程竞争的有时不是变量名，而是 cache line
- 如果两个变量恰好挤在同一条 line 里
- 一方写入就会让另一方缓存副本失效
这就是 false sharing 的典型机制。

### 例题 2 参考答案
先写 A 再写 B，不代表别的核心也按这个顺序观察到，因为：
- 编译器可能重排
- CPU 可能乱序推进
- 写入传播到其他核心也不是瞬时统一完成
这就是为什么并发程序需要同步原语和内存模型约束。

### 对比题参考讲法
- cache coherence：解决同一地址多个缓存副本何时保持一致
- memory consistency：解决多个读写操作在不同核心看来能否假定某种顺序
- 前者更偏共享数据副本协调，后者更偏并发程序可观察语义

## 常见误答

- “一致性问题只会在两个线程改同一个变量时出现。”
- “缓存一致性和内存模型只是两个翻译不同的名字。”
- “只要测试里没复现，没加同步也说明代码大概率没问题。”

## 能力判定

- 能区分 coherence 和 consistency 不是一回事

### 理解
- 能解释 false sharing 为什么会伤害性能

### 熟练
- 能把共享写、cache line、同步原语和并发正确性放到同一个模型里考虑

- 你能解释为什么不同变量也会发生 false sharing 吗？
