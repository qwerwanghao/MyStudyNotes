# 阻塞、非阻塞与事件驱动 IO

> 这篇笔记要回答：**为什么有的程序在等 I/O 时会卡住线程，有的程序却能同时管理很多连接；阻塞、非阻塞、事件驱动到底差在哪。**

## 为什么这篇很关键

服务端程序、网络程序、文件读写性能问题，
很多最后都会回到：
- 当前线程是在等吗？
- 能不能不等？
- 等的时候是谁在帮你盯事件？

## 先建立一个核心心智模型

### 区别不在“有没有数据”，而在“等待责任放在哪”
可以先这样理解：
- **阻塞**：当前线程自己停下来等
- **非阻塞**：当前线程不愿意原地等，先回来做别的
- **事件驱动**：把“盯事件”的工作系统化，让程序在事件到了再集中处理

## 一个最小案例：读 socket 时为什么行为会不同

同样是 `read/recv`：
- 阻塞模式下，没数据就挂起当前线程
- 非阻塞模式下，没数据可能立即返回
- 事件驱动模式下，程序通常先注册关注，再等系统告诉它“现在可读了”

所以差别不在 API 名字，而在：
- **线程是否原地等待**
- **等待逻辑由谁承担**

## 为什么阻塞不是原罪

阻塞模型的优点很明显：
- 直观
- 容易写
- 控制流自然

所以在：
- 简单程序
- 并发量不高
- 每个线程独占任务

这些场景下，阻塞 I/O 仍然很合理。

## 为什么非阻塞和事件驱动会流行

因为当连接数、并发数上来后：
- 让大量线程都原地等 I/O 会很浪费
- 线程切换、栈空间、调度成本都会上升

于是系统和程序更希望：
- 少一些“傻等线程”
- 多一些“有事再处理”的机制

这就是事件驱动模型存在的现实动机。

## 常见误区

### 误区 1：非阻塞一定更高级
不一定。它通常更复杂，不是所有场景都值得付这个复杂度。

### 误区 2：事件驱动就等于没有阻塞
不准确。底层某些阶段仍可能阻塞，只是等待责任和结构组织方式发生了变化。

### 误区 3：阻塞一定低效
不绝对。低并发、逻辑简单的场景下，阻塞模型可能更适合。

## 最小例题

### 例题 1：为什么高并发网络服务往往不想让“一连接一线程”一直阻塞等待？
因为这样会带来：
- 线程数量膨胀
- 调度和切换成本上升
- 空间和管理成本变大

### 例题 2：事件驱动模型最核心的收益是什么？
不是“魔法变快”，而是：
- 把等待组织得更集中
- 让程序更少依赖大量阻塞线程

## 对比题

### 阻塞 IO vs 非阻塞 IO vs 事件驱动 IO，最本质的区别是什么？
请说明：
- 谁在等
- 谁负责感知事件
- 复杂度和资源成本各自偏向哪里

## 参考答案

### 例题 1 参考答案
因为一连接一线程在高并发下会把系统推向高线程数、高切换成本和高调度压力。

### 例题 2 参考答案
事件驱动的核心收益，是把大量“等事件”的动作组织成更集中、更少线程的处理模型。

### 对比题参考讲法
- **阻塞 I/O**：线程自己原地等
- **非阻塞 I/O**：线程先回来，避免原地卡住
- **事件驱动 I/O**：让系统化的事件通知机制接管“盯可读 / 可写”这件事

## 常见误答

- 误答 1：把非阻塞直接等于高性能。
- 误答 2：把事件驱动理解成“没有等待成本”。
- 误答 3：忽略模型复杂度和场景适配。

## 能力判定

- **入门**：能区分阻塞 / 非阻塞 / 事件驱动三种模型
- **理解**：能说清它们在“等待责任”上的差异
- **熟练**：能解释为什么高并发服务偏好事件驱动
- **迁移**：能把 I/O 模型理解迁移到服务端架构与性能分析里

## 通关检查点

学完这篇后，你至少应该能做到：
- 解释三种 I/O 模型最本质的区别
- 理解为什么阻塞模型并不天然落后
- 理解事件驱动在高并发里的现实价值
