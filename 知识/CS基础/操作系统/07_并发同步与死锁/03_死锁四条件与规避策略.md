# 死锁四条件与规避策略

> 这篇笔记要回答：**为什么系统里会出现“谁都不动、谁也退不出来”的状态，以及工程上通常怎样尽量避免这种局面。**

## 为什么死锁值得单独讲

在并发里，“同步”解决的是秩序，
但同步工具一多，就可能出现另一类更糟糕的问题：
- 没有谁在出错
- 也没有谁在崩溃
- 但所有人都卡住了

这就是死锁。

## 先建立一个核心心智模型

### 死锁不是“线程慢了”，而是等待关系形成了闭环
如果：
- A 等 B
- B 等 C
- C 又等 A

那系统就进入了一个没有外力就难以前进的闭环状态。

## 为什么经典“四条件”重要

死锁不是完全随机出现的。经典上，如果同时满足这些条件，死锁就有可能发生：
1. 互斥使用
2. 占有并等待
3. 不可剥夺
4. 循环等待

这四条件重要，不是因为要背诵，
而是因为它告诉你：
> **只要破坏掉其中某一条，死锁形成的结构就会被打断。**

## 一个最小案例：两个锁为什么就够了

假设：
- 线程 A 先拿锁 L1，再等锁 L2
- 线程 B 先拿锁 L2，再等锁 L1

这时候就形成：
- A 等 B
- B 等 A

这个案例说明：
- 死锁不需要复杂系统
- 两个线程、两个锁，就足够出现最小闭环

## 为什么死锁不是只靠“程序员小心一点”就能完全解决

因为系统越复杂：
- 资源越多
- 路径越多
- 同步链越长

死锁风险就越不容易靠人工直觉完全避免。

所以工程上通常要靠策略：
- 资源顺序一致化
- 超时与回退
- 尽量减少持锁时长
- 避免在持锁时再做高风险阻塞操作

## 常见规避思路为什么有效

### 1. 固定加锁顺序
所有线程都按同一资源顺序拿锁，
可以显著减少循环等待形成的概率。

### 2. 超时 / 回退
如果等太久，就主动放弃，打破僵局。

### 3. 缩短持锁区间
锁持有越久，等待链越容易拉长。

### 4. 避免在持锁状态下做阻塞 I/O
因为这很容易把等待关系扩散到系统更多层。

## 常见误区

### 误区 1：死锁一定伴随高 CPU 占用
不一定。很多死锁反而表现为线程都睡着了，CPU 看起来不忙。

### 误区 2：只要用锁就一定会死锁
不对。锁带来死锁风险，但是否形成闭环，取决于资源获取结构。

### 误区 3：死锁和性能差是同一件事
不是。性能差可能是慢，死锁是根本不前进。

## 最小例题

### 例题 1：为什么两个线程、两个锁就足以构造最小死锁？
因为只要等待关系形成闭环，就不需要更多角色也能卡死。

### 例题 2：为什么“统一加锁顺序”常常能显著降低死锁风险？
因为它在结构上减少了形成循环等待的机会。

## 对比题

### “锁竞争严重” vs “已经死锁”，最关键的差别是什么？
请说明为什么前者是慢，后者是停。

## 参考答案

### 例题 1 参考答案
最小死锁只需要：
- A 持有 L1 等 L2
- B 持有 L2 等 L1
这样等待闭环就成立了。

### 例题 2 参考答案
统一顺序有效，是因为它在结构上尽量打断“你等我、我等你”的环形获取关系。

### 对比题参考讲法
- **锁竞争严重**：还能推进，只是慢
- **已经死锁**：等待形成闭环，系统不再推进

## 常见误答

- 误答 1：把死锁简单当成“线程卡了一会儿”。
- 误答 2：认为 CPU 不忙就不是严重问题。
- 误答 3：把规避策略理解成“经验技巧”，而不是在破坏死锁形成条件。

## 能力判定

- **入门**：能解释死锁为什么是一种等待闭环
- **理解**：能说出死锁四条件存在的意义
- **熟练**：能解释统一加锁顺序为什么有效
- **迁移**：能把死锁理解迁移到真实工程锁设计与资源申请顺序中

## 通关检查点

学完这篇后，你至少应该能做到：
- 解释什么是死锁、为什么它不是普通“慢”
- 理解四条件为什么重要
- 说清几种常见规避策略为什么有效
