# 陷入、中断与系统调用执行路径

> 这篇笔记要回答：**一次系统调用到底怎样穿过权限边界进入内核；陷入、中断、异常这些看起来相近的词，在 OS 路线上应该怎么分清。**

## 为什么这篇必须单独讲

很多人会混在一起说：
- 中断
- 异常
- 系统调用
- trap / 陷入

但在 OS 学习里，如果不先分清这几类控制流入口，后面看权限边界、I/O 事件、页故障和调度切换时就会很乱。

## 一句话先抓住核心区别

### 系统调用 / trap
- 更像程序主动请求内核介入

### 异常
- 更像当前指令触发了需要系统处理的同步事件

### 中断
- 更像来自外部设备或时钟的异步事件

它们都可能进入内核，
但进入的原因和语义并不一样。

## 一个最小过程案例：`read()` 为什么不是普通函数调用

表面上程序只是写：
- `read(fd, buf, n)`

但真实过程更像：
1. 用户态准备参数
2. 执行陷入指令 / 受控入口
3. CPU 切换到更高权限执行路径
4. 内核检查参数、对象和权限
5. 执行读请求或安排等待
6. 返回用户态，并带回结果或错误码

这个过程说明：
- 系统调用不是“调用了一个更底层函数”
- 而是**跨过权限边界进入受控执行路径**

## 为什么系统调用路径会和调度、阻塞连在一起

因为系统调用进入内核后，不一定立刻返回：
- 可能发现资源不可用
- 可能需要等待 I/O
- 可能触发睡眠和唤醒
- 甚至可能在返回前发生调度切换

所以看懂系统调用路径，才能理解：
- 为什么一个简单 API 也可能“卡住”
- 为什么线程可能在内核里被挂起

## 为什么计组里的中断知识还不够

计组更关注：
- 事件怎样打断控制流
- CPU 怎样保存和恢复现场

OS 更进一步关心：
- 这些入口怎样映射成系统服务路径
- 用户态与内核态之间怎样切换
- 处理结束后怎样安全返回

## 常见误区

### 误区 1：系统调用就是普通函数调用，只是实现更底层
不是。系统调用的关键在于权限切换和受控入口。

### 误区 2：中断、异常、trap 都差不多，反正都会进内核
不准确。它们进内核的原因不同，后续处理语义也不同。

### 误区 3：进入内核后就一定马上返回
错误。内核路径里可能阻塞、调度、等待、再恢复。

## 最小例题

### 例题 1：为什么系统调用一定要通过受控入口进入内核？
答案要点：
- 因为普通用户代码不能随便获得高权限
- 系统必须校验请求并维护边界

### 例题 2：为什么一次 `read()` 可能导致线程被挂起？
答案要点：
- 因为内核可能发现数据尚未就绪
- 当前执行流可能转入阻塞等待路径

## 对比题

### trap / 系统调用 vs 异常 vs 中断，最本质的差别是什么？
请回答：
- 事件是谁发起的
- 是同步还是异步
- 进入内核后更偏“请求服务”还是“处理事件”

## 参考答案

### 例题 1 参考答案
系统调用必须通过受控入口进入内核，因为权限提升不能由普通代码随意完成。OS 需要在入口处建立检查、切换和服务语义。

### 例题 2 参考答案
`read()` 可能导致线程挂起，因为进入内核后，如果数据还没准备好，OS 会让当前线程等待，而不是伪造一个立即完成的结果。

### 对比题参考讲法
- trap / 系统调用：程序主动请求系统服务
- 异常：当前指令触发的问题或同步事件
- 中断：外部设备 / 时钟引发的异步事件
它们都能进内核，但“为什么进去”并不相同。

## 常见误答

- “系统调用只是 libc 函数的另一种写法。”
- “只要进内核，就是同一种机制。”
- “中断一定等于发生线程切换。”
