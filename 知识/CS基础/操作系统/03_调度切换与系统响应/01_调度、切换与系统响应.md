# 调度、切换与系统响应

> 这篇笔记要回答：**CPU 时间到底是怎么在多个任务之间分配的，为什么“切一下任务”远比看上去更贵。**

## 为什么调度是 OS 的主心骨之一

只要系统里不止一个任务想跑，操作系统就必须回答：
- 谁先跑？
- 谁后跑？
- 跑多久？
- 被抢占后怎么恢复？

这就是调度问题。

## 先建立一个核心心智模型

### 调度的本质不是“排队”，而是“在有限 CPU 时间里做系统级取舍”
CPU 是有限资源。
而系统通常同时面临：
- 前台交互任务
- 后台计算任务
- I/O 等待任务
- 定时器或系统服务

调度不是简单排个序，而是在：
- 响应速度
- 吞吐量
- 公平性
- 切换成本

之间不断平衡。

## 什么叫上下文切换

上下文切换可以理解成：
- 把当前执行流的现场保存起来
- 把另一个执行流的现场恢复出来

这意味着系统至少要处理：
- PC
- 寄存器
- 栈和控制状态
- 调度信息

所以切换并不是“换个任务指针”那么简单。

## 一个直觉案例：为什么你拖动窗口时，后台任务不会把界面完全卡死

因为系统不会只让后台任务一直霸占 CPU。

它通常会：
- 给交互任务更及时的响应机会
- 抢占长期占 CPU 的后台计算
- 在多个就绪任务间轮换执行

这个例子说明：
- 调度的目标不只是“把活干完”
- 还要“让系统对人类看起来像是及时的”

## 常见调度目标到底在取舍什么

### 1. 响应时间
用户操作后多久看到反应。

### 2. 吞吐量
单位时间完成多少工作。

### 3. 公平性
是否总有某类任务被饿死。

### 4. 切换成本
切得太频繁，会把时间花在切换本身上。

## 为什么“时间片”不是万能药

给每个任务切一小段时间看起来很公平，
但时间片太小会导致：
- 切换过多
- 上下文开销显著
- 缓存和局部性被破坏

时间片太大又会导致：
- 前台响应差
- 某些任务等太久

所以调度永远是权衡，不是绝对正确答案。

## 进程切换和线程切换为什么调度上体验不同

因为切换代价不同。

- 进程切换常牵涉更大的资源和地址空间变化
- 同进程线程切换常常更轻，但仍不是零成本

这也是为什么“线程很多 ≠ 系统就一定更快”。

## 常见误区

### 误区 1：调度就是平均轮着跑
不是。调度看的是系统目标，不只是平均分时间。

### 误区 2：切换得越勤，系统越灵活
不一定。太频繁切换会让系统把很多时间浪费在切换本身。

### 误区 3：只要 CPU 忙着，系统就高效
也不对。高效不仅看 CPU 忙不忙，还看是否忙在真正有价值的工作上。

## 最小例题

### 例题 1：为什么“切换任务”本身会消耗时间？
因为系统必须：
- 保存当前上下文
- 恢复另一个上下文
- 还可能影响缓存和局部性

### 例题 2：为什么交互型任务通常不应该和大批量后台任务用完全一样的调度思路？
因为交互型任务更重视响应速度，
而后台任务更重视吞吐和总完成量。

## 对比题

### “公平” vs “响应快” vs “吞吐高”，调度到底在平衡什么？
请不要只回答名词，而是说清：
- 三者为什么经常互相拉扯
- 为什么不存在一个永远最优的统一策略

## 参考答案

### 例题 1 参考答案
任务切换不是抽象动作，它需要真实保存 / 恢复执行现场，还可能破坏当前缓存状态，所以切换本身有成本。

### 例题 2 参考答案
因为不同任务类型目标不同：
- 交互型更在乎快速响应
- 后台型更在乎稳定推进和吞吐
如果一刀切，往往会让其中一边体验很差。

### 对比题参考讲法
- **公平**：别让任务长期饿死
- **响应快**：让用户尽快看到反馈
- **吞吐高**：单位时间多完成工作
调度真正难的地方就在于：这三者常常不能同时最大化。

## 常见误答

- 误答 1：把调度理解成“平均轮流执行”即可。
- 误答 2：认为切换是零成本。
- 误答 3：认为 CPU 一直很忙就说明调度很好。

## 能力判定

- **入门**：能说出调度至少在分配 CPU 时间
- **理解**：能解释上下文切换为什么有成本
- **熟练**：能说明响应、吞吐、公平之间的取舍
- **迁移**：能把调度直觉迁移到线程设计、性能分析和系统响应问题里

## 通关检查点

学完这篇后，你至少应该能做到：
- 解释调度在解决什么问题
- 说清上下文切换为什么不便宜
- 理解为什么不同任务类型适合不同调度侧重点
