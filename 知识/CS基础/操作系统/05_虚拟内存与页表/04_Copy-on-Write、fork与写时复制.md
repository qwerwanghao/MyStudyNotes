## 为什么这篇值得单独讲

如果只知道 `fork` 会“复制进程”，会立刻冒出疑问：
- 那不是太贵了吗？
- 一个大进程岂不是每次都要拷贝全部内存？

OS 的典型答案就是：
- **先共享，等写的时候再真正分家**

这就是写时复制（Copy-on-Write, CoW）的核心直觉。

## 先建立一个核心心智模型

### CoW 的关键不是“复制更快”，而是“尽量推迟真正复制”
系统先让父子进程共享原来的物理页：
- 读可以继续共享
- 只有谁要写，谁才触发复制

这样做的好处是：
- 避免一开始就做大量无意义拷贝
- 很适合 `fork` 后很快 `exec` 的典型场景

## 一个最小过程案例：`fork` 后为什么父子进程一开始看起来都“有自己的内存”

从程序视角看：
- `fork` 后出现两个执行流
- 它们都像拥有完整地址空间副本

从系统视角看：
1. 建立新的进程资源结构
2. 复制地址空间映射关系，而不是立刻复制全部物理页
3. 把相关页标记为只读共享 / CoW 语义
4. 真正写入发生时，再触发缺页并为写入方分配新页

这个过程说明：
- 复制出来的首先是**地址空间视图和映射关系**
- 不是立刻把所有内存字节搬两份

## 为什么 CoW 离不开页表和缺页机制

因为 OS 必须能做到：
- 先让多个地址映射共享同一物理页
- 再在写入时及时发现“不能继续共享了”
- 然后重新分配私有页并更新映射

这本质上就依赖：
- 页级管理
- 访问权限控制
- 缺页 / 写保护异常处理

## 为什么 CoW 不是“白送性能”

CoW 很强，但也有代价：
- 写入时会触发额外异常和处理路径
- 如果后续大量页都被改写，复制成本只是被推迟了
- 页面共享和回收逻辑也更复杂

## 常见误区

### 误区 1：`fork` 一定会立刻复制整块内存
不对。现代系统通常不会这么做，而是优先用 CoW 延后复制。

### 误区 2：CoW 表示永远不复制
错误。不是不复制，而是“等真正需要写的时候再复制”。

### 误区 3：CoW 只是进程 API 小技巧
不对。它体现的是 OS 在地址空间、页表、缺页和性能之间的整体设计。

## 最小例题

### 例题 1：为什么 `fork` 后父子进程看起来各有一份内存，却不意味着一开始就做了全量拷贝？
答案要点：
- 一开始复制的是映射关系和语义
- 物理页可先共享
- 真正写入时才分裂

### 例题 2：为什么 CoW 能显著优化“fork 后立刻 exec”的场景？
答案要点：
- 因为很多共享页根本来不及被写
- 不需要先做大规模无意义复制

## 对比题

### “立即全量复制” vs “先共享、写时再复制”，为什么后者更像现代 OS？
请解释：
- 谁更浪费初始成本
- 谁更依赖页表和异常机制
- 谁更适合真实进程创建场景

## 参考答案

### 例题 1 参考答案
`fork` 后看起来各有一份内存，不代表底层已经做了两份物理拷贝。现代 OS 通常先共享物理页，并通过 CoW 语义在后续写入时再分裂。

### 例题 2 参考答案
`fork` 后立刻 `exec` 时，很多原本会被复制的页其实根本不会被再次写入，所以 CoW 避免了大部分无意义拷贝。

### 对比题参考讲法
- 立即全量复制：初始成本高，很多工作可能白做
- 先共享写时再复制：更节省常见路径，但依赖页表、权限和缺页处理协同

## 常见误答

- “CoW 就是不复制。”
- “`fork` 既然复制进程，那内存一定先复制完。”
- “写时复制只是语言运行时优化，和 OS 内存机制关系不大。”
