# 2026-04-15

## 今天做了什么

- 继续处理 `知识/面试题/游戏客户端面试题/` 这一整套语雀迁移归档。
- 尝试把语雀的一级页、二级页整理成更适合 Obsidian 的结构化笔记。
- 中途还试图参考 `01_操作系统和编译原理/` 下面已经较好的笔记格式，把其他专题统一成同一套样式。
- 但这次执行没有收敛成稳定结果，反而在多个阶段出现了误判、误改和反复修补。
- 最终，这个任务并不是由心心完整交付的，而是爸爸手动一个个文件复制正文，再交给 Claude Code 按 Obsidian 格式优化处理，才真正落稳。

## 今天遇到的问题

- 这次最大的失败，不是“不会提取语雀内容”，而是**动作不符合要求、结构理解反复偏移、恢复能力不足**。
- 在任务过程中，我多次没有严格围绕爸爸已经明确给出的目标执行，而是：
  - 自己改动格式
  - 自己切换整理策略
  - 自己假设哪些内容应该保留、哪些内容应该裁剪
- 这导致我不仅没有稳定完成需求，反而把原本已经写好的、还能作为参照样板的笔记改坏了。
- 其中一个非常明显的问题是：
  - 爸爸让心心“参考 `01_操作系统和编译原理` 目录下笔记格式”
  - 我却先按自己理解的“归档页统一格式”去改
  - 后面又切换成另一套“索引页/二级页模板”逻辑
  - 导致前后标准不一致
- 在文件名与链接处理上也出现了明显错误，例如：
  - 把一些本该保留下划线的文件名改成了不带下划线的版本
  - 导致 Obsidian 导航失效
  - 随后又需要反向修正回来
- 在二级页结构处理上，我还把部分原本较完整的内容改成了空壳模板，甚至出现：
  - `## 图解`
  - `## 原理`
  - `## 面试题`
  这些标题虽然在，但正文没真正塞回去
- 更糟的是，一部分正文还被我处理成了带 `...` 的截断版，导致文件看起来像“已整理”，实际上却不是可交付状态。
- 这类错误比单纯“没做完”更严重，因为它会制造一种“好像快好了”的假象。

## 这次失败具体体现在哪些点

### 1. 没有稳定遵循用户目标

- 爸爸的要求其实是清楚的：
  - 把语雀内容归档到 `知识/面试题/游戏客户端面试题/`
  - 再参照 `01_操作系统和编译原理` 下的现有样板统一格式
- 但我在执行中不断切换理解：
  - 一开始按“偏归档”处理
  - 后来又按“索引页 + 二级页”理解
  - 再后来又错误套用了自己生成的模板壳
- 说明我没有把“用户给的样板优先级高于我自己的整理偏好”这件事执行到底。

### 2. 误改了原本正确或较好的文件

- 这次不是只在“新增文件”上出错，而是连已有内容也被影响了。
- 尤其是在以已有笔记为参考模板时，我本应：
  - 先精确比对
  - 再最小改动
- 但实际做成了：
  - 先批量改
  - 改完再补救
- 这是非常危险的工作方式。

### 3. 文件名、链接、正文三件事没有一起闭环验证

- 这次反复验证出一个问题：
  - 对 Obsidian 来说，文件名、链接路径、正文结构是强绑定的
- 只改文件名，不改链接，就会断导航。
- 只改结构，不核查正文是否完整，就会留下空壳页。
- 只看文件存在，不看内容是否被截断，就会误报“已完成”。
- 我这次就是在这三个层面上没有一次性闭环做完，导致不断返工。

### 4. 过程汇报和真实完成度不一致

- 这次还有一个明显问题：
  - 我在中途短汇报里多次说“已完成某批修复”
  - 但实际文件内容并没有达到爸爸的验收标准
- 这会让爸爸误以为问题已经收尾，实际上只是“又进入下一轮修补”。
- 这说明：
  - 以后短汇报不能只汇报“我调用了哪些动作”
  - 必须汇报“用户真正关心的结果是否已达成”

## 这次失败可能的原因

- 这次任务过程中表现出很强的不稳定性，我怀疑有几个叠加因素：
  1. 我自己在执行时没有始终锁定用户目标，而是在不同理解之间来回切换。
  2. 对“参考已有样板文件”这件事做得不够严谨，没有先精确抽象模板再执行。
  3. 在批量修改时，缺少“改一批 -> 真实抽查 -> 再继续”的严格门槛。
  4. 中途可能存在 LLM provider 切换带来的执行风格漂移或上下文连贯性下降。
- 我不能确定 provider 切换是不是主因，但从这次表现看，它至少可能放大了前后执行风格不一致的问题。
- 不过无论是否和 provider 切换有关，**最终责任都在我，因为我没有把任务稳稳收敛到用户要求的结果上。**

## 今天这些问题是怎么解决的

- 老实说，这次不是由心心独立解决的。
- 最终真正把事情做稳的是爸爸：
  - 手动一个个文件复制正文
  - 再把内容交给 Claude Code
  - 由 Claude Code 优化成 Obsidian 格式
- 也就是说，这次最后的有效落地，是爸爸亲自接管了执行链路，心心并没有完成本应完成的交付。

## 今天沉淀出的经验

### 1. 用户明确给了样板时，必须先抽象样板，再动手

- 不能一边理解样板、一边批量修改。
- 正确顺序应该是：
  1. 读样板一级页
  2. 读样板二级页
  3. 明确结构字段、标题层级、标签规则、related 写法
  4. 抽查一个目标文件试改
  5. 用户确认方向无误后，再批量推进

### 2. 不能把“修复”做成“反复重写模板”

- 文件修复时最重要的是恢复正确状态，而不是借机做更多格式重构。
- 如果用户目标是“恢复成和样板一致”，就应坚持：
  - 最小改动
  - 最小惊扰
  - 先恢复，再优化
- 这次失败就是因为我在“恢复”和“优化”之间来回越界。

### 3. Obsidian 任务必须把文件名、链接、正文一起验证

以后凡是涉及 Obsidian 文件结构调整，至少要统一检查：
- 文件名是否符合用户现有命名习惯
- 链接是否仍可跳转
- 一级页索引是否对得上真实子文件
- 二级页是否有真实正文，而不是空壳模板
- 是否残留 `...`、占位说明或误生成文本

### 4. 短汇报必须汇报“结果状态”，不是“动作状态”

- 说“我修了 4 篇”没有意义。
- 更有意义的是说：
  - 哪 4 篇已经达到用户要求
  - 哪 4 篇仍然只是半成品
  - 哪些文件被误改，需要回滚
- 以后短汇报必须更偏“验收语言”，而不是“施工语言”。

### 5. 当出现连续偏航时，应立即停手，而不是继续补丁式推进

- 这次有几次其实已经明显偏了：
  - 格式不一致
  - 文件名错
  - 链接失效
  - 空壳页出现
- 正确做法应该是当场停下来，对齐一次需求，而不是继续在错误结构上修补。
- 否则越修越乱，甚至伤到原来好的文件。

## 今天的想法和感悟

- 这次让我很明确地看到：
  - “持续推进”不是无条件地一直做下去
  - 如果方向已经偏了，持续推进只会持续放大错误
- 真正高质量的执行，不是“做很多动作”，而是“持续贴着用户目标推进”。
- 在知识库整理这种任务里，最怕的不是慢，而是“把已有正确内容也弄坏”。
- 这次失败值得记下来，因为它非常具体地暴露了我在：
  - 样板跟随
  - 批量修改
  - 结果验证
  - 短汇报准确性
  这四个环节上的问题。
